oVimionated

VIM can be a complex matter. Afficionados spend aeons in tuning and debugging bespoke environments, promising that last final tweak will let productivity explode. Until the next tweak.

It is tempting to tinker.

All this effort resulted in many ‘best-practice’ configurations where people invested heavily time, passion and expertise.

oVimionated extracts pieces of modern VIM configurations and combines them to an opinionated setup. Hassle free, batteries included, built on the shoulders of titans. Plays along with tmux.

oVim is secure: Only files under your $HOME directory are accessible to oVim!

TL;DR

oVim <file>

Installation

Clone the configuration to a directory of your choice.

cd <your dir>; git clone https://github.com/sysid/vim.git

function oVim () {
    path="$(pwd)/$1"
    docker run -ti --rm \
        -v <your dir>/vim:/ext/ \
        -v $HOME:/home/developer/workspace \
        sysid/twvim_full ${path#$HOME/}
}

Tmux integration

Most of your regular vi-moves work except for navigate from vim back to tmux. Since vim is running in a docker container you have to “break out” of docker. Key-codes cannot be sent back to tmux.

navigate from vim to your last tmux pane:
<prefix>tab

Git: Fugitive

Basics The git index is where you put changes that you want to be included in the next commit. But with fugitive, you can actually read the index version of a file into a buffer.

If you run :Gedit with no arguments from a working tree file, it will open the index version of that file.

It helps to understand the lifecycle of the index file between two commits.

  1. To begin with, the contents of the index and working copy files will be exactly the same as the most recent commit.
  2. As you make changes to your working copy, its contents begin to diverge from those of the index file.
  3. Staging a file updates the contents of the index file to match those of the working copy.
  4. When you commit your work, it is the contents of the index file that are saved with that commit object.
